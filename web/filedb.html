Conflict Analysis

Write conflicts are normal and expected in multi-user database environments. ForgeFileDB serializes conflicting writes rather than letting them corrupt data. A conflict means: client A wanted to write while client B was writing — ForgeFileDB made client A wait. Zero corruption, small wait. The conflict counter is a health indicator: low = good, very high = consider migrating to MariaDB/PostgreSQL for this workload.

How ForgeFileDB Works

The Corruption Problem

File-based databases (ElevateDB, DBISAM, dBase, Access, FoxPro) open their data files directly over SMB. When two clients write simultaneously without coordination, the OS's oplock caching can serve stale data to one client while the other writes — causing partial writes that corrupt indexes and data files. ForgeFileDB coordinates these writes at the SMB level.

Lock Coordination

ForgeFileDB watches all database files via inotify and correlates open/close events with Samba session data. When client A opens a file for writing, ForgeFileDB registers an exclusive lock. Client B's conflicting write is queued until A's write flushes and closes. No changes to client software required.

Snapshot Versioning

After every batch of writes, ForgeFileDB creates a snapshot of the database directory. On btrfs, this is instant and zero-cost (copy-on-write). On other filesystems, it uses rsync to copy only database files. Snapshots are retained for 48 versions per database, with automatic pruning.

mDNS Discovery

ForgeFileDB broadcasts its presence on the local network via Avahi/mDNS as _forgeos-filedb._tcp and _edb-server._tcp. Clients on the same network can discover it without manual configuration. No proprietary protocol is implemented — this is pure service discovery.

Concurrent User Limits

With ForgeFileDB + correct SMB settings: ~20-30 concurrent users for mixed read/write workloads, 100+ for read-heavy workloads. Beyond 30 concurrent write users, migrate to MariaDB or PostgreSQL (ForgeOS includes both). ForgeFileDB can export your file-based DB to MariaDB.

Supported Databases

ElevateDB .edb/.edbt/.edbi · DBISAM .db · NexusDB .nxd · dBase/FoxPro .dbf · MS Access .mdb/.accdb · SQLite .sqlite · Firebird .fdb · TurboDB .dat · Paradox .px

Settings
Snapshot Debounce (seconds)
Minimum time between automatic snapshots for a given database directory. Prevents snapshot storms during heavy write activity.
Max Snapshots Per Database
Oldest snapshots are pruned automatically once this limit is reached. 48 = ~2 days of hourly snapshots.
Write Trigger Threshold
Number of writes to a single file before an immediate snapshot is triggered. Catches heavy write sessions.
Watch Root Path
Root directory to watch for database files. Typically the root of your Samba shares. Subdirectories are watched recursively.
